## Spacing Scale

Declared values (multiples of 4). Map to Java `LayoutManager` insets, gaps, and `EmptyBorder`.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | `EmptyBorder(4,4,4,4)` -- tight internal padding; `FlowLayout` vgap on compact bars |
| sm | 8px | Grid gap between drum pads (`GridLayout(hgap=8, vgap=8)`); gap between adjacent keyboard keys; `Box.createHorizontalStrut(8)` |
| md | 16px | `EmptyBorder(8,12,8,12)` -- standard component padding; keyboard row vertical separation |
| lg | 24px | `BorderLayout` hgap/vgap for inter-region gaps; `Container.setPreferredSize(height)` for KeyboardHintPanel content height |
| xl | 32px | Not used in Phase 3 (reserved) |
| Pad size | 48px | **Exception: PadButton dimension** -- `setPreferredSize(48, 48)`. Exactly square, fits 4 columns in 240px SidebarPanel: (4x48) + (3x8) + (2x12) = 192+24+24 = 240 |

Exceptions:
- `PadButton`: 48x48px (not a multiple of 4 on the major axis? Actually 48 IS a multiple of 4 -- 48=4x12. This is fine.)
- `KeyboardHintPanel` per-row height: 28px (a multiple of 4)
- `GridLayout` pad margins: 12px horizontal padding via `EmptyBorder(0,12,0,12)` on the grid container

## Typography

**Font family:** `"SansSerif"` (Java logical font, maps to platform UI font). Use `Font.PLAIN` for body, `Font.BOLD` for emphasis only. No more than 2 weights per phase.

| Role | `java.awt.Font` | Usage |
|------|-----------------|-------|
| PadLabel | `new Font("SansSerif", Font.PLAIN, 10)` | Drum pad name labels (e.g., "Kick", "Snare") -- rendered in the center of each 48x48 PadButton |
| KeyLabel | `new Font("SansSerif", Font.PLAIN, 11)` | Key names on KeyboardHintPanel virtual keys; Note names (e.g., "C4") on key tooltips |
| Body | `new Font("SansSerif", Font.PLAIN, 12)` | JPopupMenu items (right-click pad menu); default JLabel text throughout UI |
| Title | `new Font("SansSerif", Font.BOLD, 14)` | Section headers if used (e.g., "DRUM PADS" label above grid) |

**Line height:** Not applicable in Java Swing -- `JLabel` and `JButton` use single-line text rendering. Multi-line text uses `JTextArea` (none in Phase 3).

**Anti-aliasing:** All custom-painted components MUST enable `RenderingHints.KEY_TEXT_ANTIALIASING` with `VALUE_TEXT_ANTIALIAS_ON` before drawing strings (follows `PianoRollPanel` pattern).

### Empty State

DrumPadGrid: No empty state. All 16 pads are always populated with defaults (D-05). The grid is never empty.
KeyboardHintPanel: Never empty. Three rows always render with the fixed key mapping from D-01. Unloaded state shows all keys idle.

### Error State

MIDI Send Failure: If a pad press or key press fails to send MIDI (e.g., synthesizer not available):
- Silent failure -- do NOT show error dialogs for individual note triggers (would spam during performance).
- Log to `System.err` only: `"MIDI send failed: note=X channel=Y"`
- Synthesizer availability is guaranteed at startup (App.java exits with error if synth can't initialize).

### Destructive Actions

| Action | Trigger | Confirmation |
|--------|---------|-------------|
| Reset pad to default | Right-click menu "Reset to {label}" | `JOptionPane.showConfirmDialog`: "Reset pad '{current_label}' to default sound '{default_label}'?" with Yes/No. No undo. |

No other destructive actions in Phase 3. Pad assignments are in-memory only, no file persistence.

### Primary Interaction Labels

- KeyboardHintPanel tooltip (per key): `"{physical_key} → {note_name} ({midi_note})"` -- e.g., `"Q → C4 (60)"`
- PadButton tooltip: `"{label} (Note {midiNote})"` -- e.g., `"Kick (Note 36)"`
- JPopupMenu items: `"{sound_name} ({midi_note})"` -- e.g., `"Bass Drum 1 (36)"`

## Interaction Contract

### Keyboard Input Flow (D-03)

1. `KeyboardFocusManager.getCurrentKeyboardFocusManager().addKeyEventDispatcher(e -> ...)` registers global key listener in `ReBeatBoxWindow.wireEngine()`.
2. On `KEY_PRESSED`: lookup keyCode in `KeyboardMapper` table. If mapped and not already in `activeKeys[note]`:
   - `receiver.noteOn(note, 100)` on Channel 0
   - `eventBus.fireLiveNoteOn(note)`
   - `activeKeys[note] = true`
   - `KeyboardHintPanel.highlightKey(keyLabel, true)`
3. On `KEY_RELEASED`: lookup keyCode. If mapped and `activeKeys[note]`:
   - `receiver.noteOff(note)` on Channel 0
   - `eventBus.fireLiveNoteOff(note)`
   - `activeKeys[note] = false`
   - `KeyboardHintPanel.highlightKey(keyLabel, false)`
4. On window focus lost: call `keyboardHintPanel.clearAllHighlights()` + `activeKeys` reset to prevent stuck highlights.
5. Key repeat events (OS auto-repeat): suppressed by `boolean[128]` dedup array per D-03.

### Drum Pad Input Flow

1. `PadButton` extends `JButton`. Mouse press on EDT triggers `noteOn(midiNote, 100)` on Channel 10.
2. Mouse release triggers `noteOff(midiNote)` on Channel 10.
3. Visual: 200ms `Timer` holds "pressed" visual state after mouse release to ensure the glow is visible even for very short clicks.
4. Right-click: `JPopupMenu` shown at mouse position. Selecting a sound calls `setMidiNote(newNote, newLabel)`. "Reset to Default" shows `JOptionPane.showConfirmDialog`.
5. Pad assignments stored in `DrumPadGrid` as `int[16]` and `String[16]` arrays -- in-memory only.

### Event Bus Extension (D-10)

```
// Added to existing NoteEventListener interface (or new LiveNoteEventListener)
@FunctionalInterface
public interface LiveNoteEventListener {
    void onLiveNoteOn(int note, int velocity);
}

@FunctionalInterface
public interface LiveNoteOffEventListener {
    void onLiveNoteOff(int note);
}

// Added to NoteEventBus:
public void subscribeLiveNoteOn(LiveNoteEventListener listener)
public void subscribeLiveNoteOff(LiveNoteOffEventListener listener)  
public void fireLiveNoteOn(int note, int velocity)
public void fireLiveNoteOff(int note)
```

PianoRollPanel subscribes to both for real-time note flash visualization (visual indicator only -- does not affect MIDI data).

### KeyboardHintPanel Repaint Strategy

Custom-painted (not component-based):
- Default state: repaint only on resize or highlight change (NOT on a timer)
- On highlight change (key pressed/released): call `repaint()` on EDT -- single-pass, targets only the affected key region
- No animation loop -- Phase 4 will add glow transitions. Phase 3 uses instant color swap.
- Panel height: `setPreferredSize(new Dimension(width, 84))` -- 3 rows x 28px per row

### KeyboardHintPanel Layout Geometry

Three rows, full width. Each row is 28px tall. Total panel height: 84px.

```
Row 1 (top):    Number row -- keys [1] through [=]        → notes C5-B5 + C6
Row 2 (middle): QWERTY row -- keys [Q] through []]         → notes C4-B4 + C5
Row 3 (bottom): Bottom row -- keys [Z] through [/]         → notes C3-B3
```

Each white key occupies one column; black keys sit between/overlap adjacent white keys. The ratio: white key width = available width / whites-per-row; black key width = 60% of white key; black key height = 70% of row height, centered vertically.

Row 1 whites: 8 keys (1,2,3,4,5,6,7,8,9,0,-,= -- wait, let me count from the D-01 mapping... 1=C5, 2=C#, 3=D5, 4=D#, 5=E5, 6=F5, 7=F#, 8=G5, 9=G#, 0=A5, -=A#, = =B5/C6... so actually it's a bit more complex because we have 12 physical keys mapping to 13 chromatic notes for the octave+1. The last key (=) maps to C6 which is the start of the next octave.)

Actually re-examining D-01: 
- Row 1 (number row) has 12 physical keys (0 through = or Shift+0 through = on some layouts, but standard US keyboard digits row) mapping from C5 to C6 (13 semitones with the last = being C6). Let me just note: the visual layout shows 12 keys per row (the actual physical keys that are mapped). Black keys are the sharps/flats between white keys.
- For painting: lay out all physical keys in order, white keys fill the row width, black keys overlay at reduced width/height.

The exact pixel math is implementation detail for the executor. The contract specifies: white keys use row height, black keys are 60% width x 70% height, centered over their column.
