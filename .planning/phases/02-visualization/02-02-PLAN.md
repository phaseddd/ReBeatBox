    **Coordinate system (implement as private helper methods):**
    
    The canvas has two zones:
    - Left zone [0, KEYBOARD_WIDTH): mini keyboard
    - Right zone [KEYBOARD_WIDTH, getWidth()): note waterfall
    
    Vertical layout:
    - `triggerLineY()` = `(int)(getHeight() * (1.0 - TRIGGER_LINE_RATIO))` — trigger line is 18% from bottom.
    - Area ABOVE trigger line = future notes (preview). Area BELOW = past notes (dimmed).
    
    Time-to-Y mapping:
    - `timeToY(long noteTimeMicros, long currentPosMicros)`:
      - `delta = noteTimeMicros - currentPosMicros` (positive = future, negative = past)
      - If delta >= 0: `y = triggerLineY - (int)(delta * pixelsPerMicroAbove)`
        where `pixelsPerMicroAbove = (double)triggerLineY / PREVIEW_MICROS`
      - If delta < 0: `y = triggerLineY + (int)(-delta * pixelsPerMicroBelow)`
        where `pixelsPerMicroBelow = (double)(getHeight() - triggerLineY) / PAST_MICROS`
    
    Pitch-to-X mapping (adaptive range, per D-02):
    - `pitchToX(int pitch)`:
      - Padding of 1 semitone on each side (Claude's discretion).
      - `effectiveMin = max(0, noteRangeMin - 1)`
      - `effectiveMax = min(127, noteRangeMax + 1)`
      - `numColumns = effectiveMax - effectiveMin + 1`
      - `colWidth = (double)(getWidth() - KEYBOARD_WIDTH) / numColumns`
      - `x = KEYBOARD_WIDTH + (pitch - effectiveMin) * colWidth`
    
    Note range is computed in Task 3 when pre-scan data arrives. For now, use default range [21, 108] (full piano). The adaptive update is trivial in Task 3.

    **Mini keyboard rendering (paintComponent helper):**
    Draw in paintComponent BEFORE note rendering so notes overlay the keyboard zone.

    - White key width = `KEYBOARD_WIDTH - 1` (leave 1px gap). Fill: dark gray (e.g., `new Color(0x2a2a2a)`). Stroke: slightly lighter gray (0x3a3a3a).
    - Black key width = `KEYBOARD_WIDTH * 0.6`. Fill: darker (0x1a1a1a). Draw over white keys (z-order).
    - Only render keys whose pitch is within the adaptive note range (skip keys outside range).
    - For white keys: draw all. For black keys: draw only those adjacent to rendered white keys.
    - Draw subtle octave separator lines at C notes (pitch % 12 == 0) — thin horizontal line across keyboard zone at the Y position matching that note's current screen position (use timeToY, same as notes). This ties the keyboard to the waterfall view.

    Extend PianoRollPanel with the full note rendering pipeline. All work is additions to the paintComponent method and new private helper methods.

    **Three-layer compositing architecture (D-08):**

    **Layer 1 (Background):** Already implemented in Task 1 — pure `Color.BLACK` fill. This layer is repainted every frame but it's just one fillRect call — negligible cost.

    **Layer 2 (Blurred notes):**
    For each RenderNote within the visible time window, produce a per-note glow:

    ```
    For each visible RenderNote note:
      1. Create BufferedImage for this single note bar (TYPE_INT_ARGB, size = bar bounds)
      2. Draw filled rectangle on the BufferedImage using NoteColorMapper.forPitch(note.pitch)
      3. Apply GaussianBlur via ConvolveOp with a 5x5 kernel, sigma ~2.0
         (Claude's discretion: kernel size 3-7, sigma 1.5-3.0 per D-07 user preference for quality)
      4. Set composite alpha based on position vs trigger line (D-06):
         - Note's Y entirely above trigger line → AlphaComposite.SrcOver.derive(1.0f) [full brightness]
         - Note's Y entirely below trigger line → AlphaComposite.SrcOver.derive(0.4f) [dimmed per D-06]
         - Note spans trigger line → split into two segments (above and below), render each with appropriate alpha
      5. Composite the blurred BufferedImage onto the main Graphics2D at the note's x,y position
    ```

    **Note bar geometry (D-04):**
    - Each note occupies one column (per semitone pitch).
    - `barX = pitchToX(note.pitch)`
    - `barY = timeToY(note.startMicros, currentPos)` (top of bar — note start time)
    - `barBottomY = timeToY(note.endMicros, currentPos)` (bottom of bar — note end time)
    - `barHeight = barBottomY - barY` (clamp to >= 2px minimum, so very short notes are still visible)
    - `barWidth = pitchToX(note.pitch + 1) - pitchToX(note.pitch)` (column width from pitch grid)
    - Subtract 1px from barWidth for a tiny gap between adjacent columns (helps visual separation).
    
    **Chords (D-04):** Multiple notes with the same startMicros naturally render as adjacent bars at the same Y position — no special handling needed. The pitch grid spacing handles this automatically.

    **Layer 3 (Foreground — drawn directly on Graphics2D, no BufferedImage):**

    1. **Grid lines (subtle):** Optional — Claude's discretion per the "pure black canvas, notes themselves define visual space" aesthetic. If included, draw as very faint horizontal lines at one-beat intervals (based on BPM) using `new Color(255, 255, 255, 15)` (near-invisible white). This helps orientation without violating D-07's spirit.

    2. **Trigger line (D-12):**
       - `int ty = triggerLineY()`
       - Draw the glow first: a wider, semi-transparent cyan stroke.
         ```java
         g2d.setStroke(new BasicStroke(8.0f, BasicStroke.CAP_ROUND, BasicStroke.JOIN_ROUND));
         g2d.setColor(new Color(0, 255, 255, 60)); // cyan, low alpha for glow
         g2d.drawLine(KEYBOARD_WIDTH, ty, getWidth(), ty);
         ```
       - Draw the core line: 2px solid cyan/white.
         ```java
         g2d.setStroke(new BasicStroke(2.0f, BasicStroke.CAP_ROUND, BasicStroke.JOIN_ROUND));
         g2d.setColor(new Color(200, 255, 255, (int)(255 * pulseAlpha))); // pulseAlpha from Task 3
         g2d.drawLine(KEYBOARD_WIDTH, ty, getWidth(), ty);
         ```
       - The glow effect uses a simple multi-pass stroke (wider translucent + narrower bright) rather than a separate BufferedImage+GaussianBlur for the line. This is cheaper and sufficient for a thin line. The pulse animation variable `pulseAlpha` is computed in Task 3; for this task, use a constant 1.0f and replace in Task 3.

    **Viewport culling (D-11):**
    Before rendering, filter renderNotes to only those within the visible window:
    - Lower bound: `currentPos - PAST_MICROS` (notes whose startMicros is after this)
    - Upper bound: `currentPos + PREVIEW_MICROS` (notes whose startMicros is before this)
    - Use `Collections.binarySearch()` on the startMicros-sorted list to find the first visible note index. Then iterate forward until notes exceed the upper bound.
    - Implementation:
      ```java
      long windowStart = currentPos - PAST_MICROS;
      long windowEnd = currentPos + PREVIEW_MICROS;
      // Binary search for first note with startMicros >= windowStart
      int firstIdx = binarySearchFirst(renderNotes, windowStart);
      // Iterate from firstIdx while note.startMicros <= windowEnd
      ```
    - The binarySearchFirst helper uses `Collections.binarySearch()` with a sentinel RenderNote. If the exact match isn't found, use `-(insertionPoint + 1)` to get the first element >= target.

    **Performance considerations (PITFALLS.md #3: Graphics2D performance):**
    - Per-note BufferedImage creation every frame is the primary cost. Cache note bar images? For v1 per D-07 user preference ("visual quality prioritized over performance"), create fresh BufferedImages each frame. The viewport culling limits this to typically 20-100 notes per frame, which is manageable.
    - Use `RenderingHints.KEY_RENDERING = VALUE_RENDER_SPEED` (not quality) since we're doing our own blur.
    - The ConvolveOp is the heaviest operation. A 5x5 kernel on small rects (~20-50px each) should be fast enough.
    - If performance profiling shows issues, Claude's discretion to reduce kernel to 3x3 or increase culling aggressiveness (1.5s instead of 2s).

    **Part A — PianoRollPanel: Timer-driven animation loop (D-13)**

    Add fields:
    ```java
    private Timer animationTimer;
    private long currentPositionMicros = 0;
    private float pulsePhase = 0.0f; // for trigger line pulse animation (0.0 to 2*PI)
    ```

    Add/update methods:
    ```java
    public void setController(PlaybackController controller) {
        this.controller = controller;
        startAnimationLoop();
    }
    
    private void startAnimationLoop() {
        if (animationTimer != null && animationTimer.isRunning()) return;
        // javax.swing.Timer runs callbacks on EDT — no manual invokeLater needed
        animationTimer = new Timer(16, e -> { // 16ms ≈ 62.5fps, close enough to 60
            if (controller == null) return;
            
            // Per D-13: directly poll controller position every frame
            currentPositionMicros = controller.getMicrosecondPosition();
            
            // Per D-14: timer continues even when paused, but position freezes
            // (controller.getMicrosecondPosition() returns static value when paused)
            
            // Update pulse animation phase (D-12)
            pulsePhase += 0.1f; // incremental phase per frame
            if (pulsePhase > 2 * Math.PI) pulsePhase -= 2 * Math.PI;
            
            repaint(); // queues a paintComponent call on EDT
        });
        animationTimer.setInitialDelay(0);
        animationTimer.start();
    }
    ```

    **Trigger line pulse animation (D-12):**
    Update the trigger line drawing in paintComponent (Task 2's code) to use a computed pulse alpha:
    ```java
    // Sine-based pulse between 0.80 and 1.00
    float pulseAlpha = 0.80f + 0.20f * (float)((Math.sin(pulsePhase) + 1.0) / 2.0);
    ```
    (Claude's discretion: linear vs sine easing — sine is smoother and more "heartbeat-like" per D-12.)

    Pass `pulseAlpha` to the trigger line color alpha. The core line uses full computed alpha; the glow halo uses alpha * 0.25 for a softer effect.

    **Currently-sounding notes (D-15):**
    Add a method to determine which notes are "sounding" at the current position:
    ```java
    public Set<Integer> getSoundingNotes() {
        // Per D-15: binary search the pre-scanned RenderNote list
        // A note is "sounding" if currentPositionMicros is within [startMicros, endMicros)
        Set<Integer> result = new HashSet<>();
        if (renderNotes.isEmpty() || controller == null) return result;
        
        long pos = currentPositionMicros;
        // Binary search for first note where endMicros > pos
        // (i.e., note hasn't ended yet)
        for (RenderNote note : renderNotes) {
            if (note.endMicros() <= pos) continue; // ended
            if (note.startMicros() > pos) break;   // not started yet (beyond current pos)
            result.add(note.pitch());
        }
        return result;
    }
    ```
    This method exists for Phase 4 (ParticleSystem) and Phase 3 (visual feedback for live notes). It does NOT depend on NoteEventBus per D-15.

    Note: The linear scan above is correct because we break after passing the current position. For large files, a proper binary search for the start index would be better, but the early-break makes this O(visible_notes) which is fine.

    **Pre-scan integration:**
    ```java
    public void onFileLoaded() {
        if (controller == null) return;
        Sequence seq = controller.getSequencer().getSequence();
        if (seq == null) return;
        
        // Pre-scan the MIDI data per D-10
        renderNotes = MidiPreScanner.scan(seq);
        
        // Compute adaptive note range per D-02
        if (!renderNotes.isEmpty()) {
            noteRangeMin = 127;
            noteRangeMax = 0;
            for (RenderNote note : renderNotes) {
                if (note.pitch() < noteRangeMin) noteRangeMin = note.pitch();
                if (note.pitch() > noteRangeMax) noteRangeMax = note.pitch();
            }
            // Add 1-semitone padding on each side (Claude's discretion)
            noteRangeMin = Math.max(0, noteRangeMin - 1);
            noteRangeMax = Math.min(127, noteRangeMax + 1);
        }
        
        // Reset position tracking
        currentPositionMicros = 0;
        
        // Ensure timer is running
        startAnimationLoop();
        
        repaint();
    }
    ```
