  <name>Task 1: Create RenderNote record and MidiPreScanner</name>
  <files>src/main/java/com/rebeatbox/visual/RenderNote.java, src/main/java/com/rebeatbox/visual/MidiPreScanner.java</files>
  <behavior>
    - RenderNote: Java record (or final class) with fields pitch(int 0-127), startMicros(long), endMicros(long), velocity(int 0-127), channel(int 0-15). Must be immutable. Provide a compact toString() for debugging.
    - MidiPreScanner.scan(Sequence): returns empty list for null/empty Sequence. Returns sorted List<RenderNote> by startMicros ascending.
    - Track 0 (conductor metadata) is skipped entirely per D-09.
    - For each remaining track: iterate events linearly. For each NOTE_ON with velocity > 0: store pending (pitch+channel → tick). For NOTE_OFF or NOTE_ON with velocity==0 on same pitch+channel: close the pair, create RenderNote. Convert tick to microseconds using Sequence resolution and tempo.
    - Orphaned NOTE_ON (no matching NoteOff by track end): set endMicros to Sequence.getMicrosecondLength() (the file's total duration).
    - Must handle Type 0 (single track) and Type 1 (multi-track) MIDI files.
    - Must handle duplicate NoteOn without NoteOff by treating the second NoteOn as an implicit NoteOff for the first.
  </behavior>
  <action>
    <!-- Implementation after tests pass -->

    **RenderNote.java:**
    Create as a Java `record` (Java 17): `public record RenderNote(int pitch, long startMicros, long endMicros, int velocity, int channel)`. Records are immutable by design. Add bounds validation in compact constructor: pitch 0-127, velocity 0-127, channel 0-15, startMicros >= 0, endMicros >= startMicros (throw IllegalArgumentException on violation). Include a toString() that shows pitch name (e.g., "C4", "A#3") plus time range for debugging.

    **MidiPreScanner.java:**
    Single static method: `public static List<RenderNote> scan(Sequence sequence)`.
    
    Tick-to-microsecond conversion: use `sequence.getResolution()` (ticks per quarter note) and a default tempo of 500,000 microseconds per beat (120 BPM). For variable-tempo files, extract tempo from MetaMessages (type 0x51) on each track to build a tempo map. This is critical for correct note durations — many MIDI files have tempo changes.

    Algorithm:
    1. Guard: if sequence == null, return Collections.emptyList().
    2. Collect tempo events: iterate all tracks (including track 0), extract MetaMessage type 0x51 (tempo). Build a sorted list of (tick, microsPerBeat) pairs. Use 500,000 as default for tick 0 if not present.
    3. Track iteration: `Track[] tracks = sequence.getTracks()`. Skip index 0 per D-09. For each remaining track:
       - Map `String key = pitch + ":" + channel` → pending NoteOn tick.
       - Iterate `track.size()` events via `track.get(i)`.
       - For ShortMessage NOTE_ON (command 0x90) with velocity > 0:
         - If key already has a pending entry, close the pending one (implicit NoteOff at this tick), add RenderNote.
         - Store new pending entry.
       - For ShortMessage NOTE_OFF (command 0x80) or NOTE_ON with velocity == 0:
         - Look up key. If found, create RenderNote with duration = tickToMicros(currentTick) - tickToMicros(pendingTick). Remove from pending.
    4. After track iteration: for any remaining unclosed pending entries, set endMicros to `sequence.getMicrosecondLength()`.
    5. Sort the result list by `startMicros` ascending using `List.sort(Comparator.comparingLong(RenderNote::startMicros))`.
    6. Return the sorted list.

    Tick-to-microsecond helper: given a tick, walk the tempo map to compute cumulative microseconds.
  </action>

  <files>src/main/java/com/rebeatbox/visual/NoteColorMapper.java</files>
  <behavior>
    - forPitch(int midiNote) returns a java.awt.Color for any valid MIDI note (0-127).
    - Low pitches (0-47) → purple to blue range. Middle pitches (48-71) → cyan to green to yellow. High pitches (72-127) → orange to red. Rainbow gradient across full range.
    - Colors should be "neon" — high saturation (>= 0.7) and brightness (>= 0.8) in HSB space.
    - Adjacent semitones should have perceptibly different hues (approximately 3-4 degrees of hue difference).
    - Same octave notes should have visually related but distinguishable colors.
    - Invalid input (< 0 or > 127) should throw IllegalArgumentException.
  </behavior>
  <action>
    <!-- Implementation after tests pass -->
    Create a utility class with one static method: `public static Color forPitch(int midiNote)`.

    Use `Color.getHSBColor()` for the mapping:
    - Map pitch 0→127 onto hue range 0.70→0.0 (wrapping through purple/blue/green/yellow/orange/red).
    - Formula: `hue = 0.70f - (pitch / 127.0f) * 0.70f`. This gives low=0.70 (purple), mid≈0.35 (green), high≈0.0 (red).
    - Saturation: 0.85f (vivid, neon feel).
    - Brightness: 0.95f (near-full brightness, looks luminous on black background).

    The hue range 0.70→0.0 passes through: purple (0.70) → blue (0.60) → cyan (0.50) → green (0.35) → yellow (0.17) → orange (0.08) → red (0.0). This is the standard rainbow spectrum from low to high.

    Validate input: throw IllegalArgumentException for midiNote < 0 or > 127.

    Claude's discretion (D-05 implementation choice): exact hue endpoints and saturation values. The formula above is a solid starting point consistent with the user's stated preference for "low=purple/blue, high=red/orange."

    This class has zero dependencies on other project packages. It only imports java.awt.Color.
